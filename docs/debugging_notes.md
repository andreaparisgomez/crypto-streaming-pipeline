# Debugging Notes

## Spark Kafka connector configuration

### Issue

PySpark failed with:

```text
AnalysisException: Failed to find data source: kafka
```

### Root Cause

The Spark session was created without the Kafka connector package.

### Fix

Configured Spark with:

```python
.config(
    "spark.jars.packages",
    "org.apache.spark:spark-sql-kafka-0-10_2.13:4.1.1"
)
```

### Notes

Using connector version `4.0.1` caused compatibility issues.  
Changing to `4.1.1` resolved the problem.

### Lesson

Spark Structured Streaming requires explicit Kafka connector dependencies.

## Unsupported streaming window functions

### Error

```text
NON_TIME_WINDOW_NOT_SUPPORTED_IN_STREAMING
```

### Root Cause

Structured Streaming does not support arbitrary SQL-style window functions such as:

```python
lag()
```

on streaming DataFrames.

### Original Attempt

Attempted logic:
- `partitionBy("coin")`
- `orderBy("timestamp")`
- `lag(previous_price)`

### Architectural Realisation

Streaming systems cannot freely access arbitrary historical rows without explicitly managing state.

### Decision

Removed lag/pct_change logic from the streaming processor and moved towards time-window aggregations.

### Lesson

Streaming Spark is fundamentally different from static Spark processing.

## PostgreSQL consumer schema mismatch

### Error

```text
KeyError: 'window_start'
```

### Root Cause

The consumer was reading older Kafka messages generated by a legacy Python processor using a previous schema.

### Old Schema

```json
{
    "coin",
    "price",
    "pct_change"
}
```

### New Spark Schema

```json
{
    "coin",
    "window_start",
    "window_end"
}
```

### Fix

Changed:

```python
auto_offset_reset="earliest"
```

to:

```python
auto_offset_reset="latest"
```

### Lesson

Streaming consumers must account for schema evolution and historical topic data.


### Understanding

- processed offsets

- recovery metadata

## Airflow environment variable propagation debugging

Issue:
Airflow DAGs failed after replacing hardcoded PostgreSQL credentials with environment variables.

Cause:
The Airflow Docker containers could not access the PostgreSQL environment variables because:
- variables existed locally inside `.env`
- but were not passed into the Docker Compose container environment

Important realisation:
A local `.env` file does NOT automatically make variables available inside running containers.

Fix:
Added PostgreSQL variables into the shared Airflow Docker Compose environment:

```yaml
environment:
  &airflow-common-env
  POSTGRES_HOST: ${POSTGRES_HOST}
  POSTGRES_DB: ${POSTGRES_DB}
  POSTGRES_USER: ${POSTGRES_USER}
  POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
```

Then restarted the Airflow services.

Additional debugging:
Initial YAML syntax errors occurred because the Docker Compose anchor:

```yaml
&airflow-common-env
```

was accidentally placed incorrectly during editing.

Final outcome:
- Airflow DAGs successfully accessed PostgreSQL credentials through environment variables
- hardcoded credentials were removed from DAG scripts
- repository became safe for public GitHub upload
