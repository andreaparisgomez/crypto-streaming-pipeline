# Debugging Notes

## Spark Kafka connector configuration

### Issue

PySpark failed with:

```text
AnalysisException: Failed to find data source: kafka
```

### Root Cause

The Spark session was created without the Kafka connector package.

### Fix

Configured Spark with:

```python
.config(
    "spark.jars.packages",
    "org.apache.spark:spark-sql-kafka-0-10_2.13:4.1.1"
)
```

### Notes

Using connector version `4.0.1` caused compatibility issues.  
Changing to `4.1.1` resolved the problem.

### Lesson

Spark Structured Streaming requires explicit Kafka connector dependencies.

## Unsupported streaming window functions

### Error

```text
NON_TIME_WINDOW_NOT_SUPPORTED_IN_STREAMING
```

### Root Cause

Structured Streaming does not support arbitrary SQL-style window functions such as:

```python
lag()
```

on streaming DataFrames.

### Original Attempt

Attempted logic:
- `partitionBy("coin")`
- `orderBy("timestamp")`
- `lag(previous_price)`

### Architectural Realisation

Streaming systems cannot freely access arbitrary historical rows without explicitly managing state.

### Decision

Removed lag/pct_change logic from the streaming processor and moved towards time-window aggregations.

### Lesson

Streaming Spark is fundamentally different from static Spark processing.

## PostgreSQL consumer schema mismatch

### Error

```text
KeyError: 'window_start'
```

### Root Cause

The consumer was reading older Kafka messages generated by a legacy Python processor using a previous schema.

### Old Schema

```json
{
    "coin",
    "price",
    "pct_change"
}
```

### New Spark Schema

```json
{
    "coin",
    "window_start",
    "window_end"
}
```

### Fix

Changed:

```python
auto_offset_reset="earliest"
```

to:

```python
auto_offset_reset="latest"
```

### Lesson

Streaming consumers must account for schema evolution and historical topic data.

## Spark checkpointing

### Observation

Checkpoint directories were created using:

```python
.option("checkpointLocation", PATH)
```

### Understanding

Checkpoint directories store:
- processed offsets
- streaming state
- recovery metadata

They do not store the actual Kafka event data.

### Lesson

Kafka stores event data.  
Spark checkpoints store processing state.

## Airflow DAG not appearing

### Issue

The custom DAG did not appear in the Airflow UI.

### Root Cause

Docker was mounting an outdated Airflow project directory after the `airflow/` folder was moved.

### Fix

Updated:

```env
AIRFLOW_PROJ_DIR=/Users/andrea/crypto-streaming-pipeline/airflow
```

Then restarted containers:

```bash
docker compose down
docker compose up -d
```

### Lesson

Docker volume mounts must point to the correct local filesystem path.
